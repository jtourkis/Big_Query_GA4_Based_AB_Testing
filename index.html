E-Commerce Experiment Code

Goal: The goal of this notebook is to recreate steps necessary to run a basic experiment on conversion rates. For the sake of realism, the notebook processes Google Analytics 4 event data directly from Google Big Query. The public event data comes from the Google Merchandise e-commerce website at https://shop.googlemerchandisestore.com between 11/1/2020 and 1/31/2021.

For simplicity, the code will evaluate one main metric, purchase conversion rate. In a real experiment, it would be important to discuss the benefits and downsides of chosen metrics. For example, low frequency events like purchase rate require extremely large sample sizes or must detect very large effects.

The proposed experiment page (a.k.a. evaluation page) will be a landing page hosting apparel named 'Apparel | Google Merchandise Store.' I've chosen to evaluate a 1 month window from 12/1 to 12/28. When possible, it is recommended that full weeks be evaluated.

Overview:

  1. Review Historical Activity for Chosen Metric (II(a)-(b))

  2. Calculate Required Sample Size (II(c))

  3. Create Table with Covariates for Balance Check (III)

  4. Run Experiment Result Checks like Balance (V)

  5. Evaluate Fake Experiment Group Results (VI)

  6. Logistic Regression Set Up (VII)

Notes:

  1. In a real experiment, there would likely be a number of secondary and guardrail metrics. If you test revenue, it is usually best practice to select at least one user retention based guardrail metric. Short term revenue growth is not worth losing your most dedicated visitors.

  2. Be careful how you interpret Secondary metrics. If we look at everything under the sun, there is a high likelihood you will find something that looks like an impact. To account for this, in statistics, if more than one primary metric is tested, confidence intervals are lengthened by multiple comparison adjustments like Benjamini and Hochberg. This may change your statistically significant result to not significant. If you discover something interesting in secondary metrics, best practice is to retest it as a primary metric.

Conclusions: For the purpose of this example, given limited data, control group conversion rates for Minimum Detectable Effect (MDE) sample size calculations are estimated based on the last non-holiday month to be .8%. ie January 2021.|

Notes: There are many ways to estimate control page visit conversion rates for minimal detectable effects in sample size calculations. If you have fairly stable conversion rates, a fairly naive method involves simply taking the last month while avoiding months with holidays or unusual activity. Alternatively, we can get a pretty good idea of expected conversion rates by taking the rate of growth YoY of the prior month times the last years estimate for the experiment's month.

Either way, its helpful to review at least 14 months to two years of time series data to see if estimates are historically reasonable.

CAUTION: 1) Averaging many months of historical data can be misleading. Changing trends can dramatically influence more recent rates. 2) When picking the effect size you want to test for, it is important to consider what effect size you want to claim victory for. It is not good practice to claim effects smaller than your Minimum Detectable Effect size were statistically significant.

Conclusion: For developed websites, the goal should be to evaluate an effect sizes between 1% to 5% with at least .8 power. However, for this very low conversion rate, a 5% lift is impractical and has very low power at reasonable sample sizes. This leaves us with two options. a) Choose a higher effect size and miss out on small impacts. b) Attempt to find a metric correlated with the conversion rate of interest but with a higher estimated control rate.

Note:

  1. Whatever effect size (MDE) is chosen, it must be reasonable in light of historical experiment effects. Running under powered tests (tests without sufficient sample size) is known to inflate statistically significant effect size estimates.

  2. Unless experiment is run on logged in users only, time periods over which visitor activity can be linked together continue to shrink due to browser changes. Running experiments for longer than 1 month, risks double exposure to treatment and control for repeat visitors where tracking lapsed.

  3. For smaller sample sizes and tiny conversion rates, traditional Wald confidence interval calculations may need to be adjusted to remain at 95%.

Step 4: Calculate required sample size for 23% Lift MDE.

Required n Per Group (n) = $\frac{(\frac{\alpha}{2}_{crit} + 1-\beta_{crit})^2*2\sigma^2}{\delta^2} $

where $\delta$ is the absolute Minimum Detectable Effect size.

Note: In some implementations, unequal groups resluts in increasing one $\sigma^2$ by a ratio of control / variant groups. See Georgiev at 11.

$\sigma_{pooled}=\sqrt{p_1-(1-p_1+p_2-(1-p_2)*Ratio}$

Goal: This section explores calculating commonly tracked features associated with e-commerce experiments. If there are strong correlations to the outcome of interest (ie conversion rate), features might be used to identify potential imbalances from bad randomizations or in a regression to remedy minor imbalances / increase precision. For a feature to be useable, it must accrue before exposure to the experiment. For continuous features, they should be uncorrelated with each other and linearly related to continuous outcome metrics.

Note:

  1. Statistically insignificant results does not mean there was no effect. A statistically insignificant result means we cannot distinguish between true observed differences in conversion rates and those relating to the variability in our data. The effect could be negative or positive. Confidence intervals tell us the predicted range.

  2. 80% power means that if there was a true effect and we followed sample size requirements. We would expect to pick it up 80% of the time.

  3. To minimize false positive rates from running many online experiments, some practitioners like Ron Kohavi suggest re-running experiments with a p-value between .01 and .05.
